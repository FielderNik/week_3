# Инструкции по репозиторию

## Контекст

Этот репозиторий относится ко второй неделе курса по AI-инструментам. Каждая неделя курса должна жить в отдельном репозитории. Каждая ежедневная задача должна жить в отдельной папке: `task_6`, `task_7`, `task_8` и так далее.

Задания этой недели начинаются с `task_6`. Папки задач могут переиспользовать общий код из этого репозитория, если это уместно.

## Общий шаблон чата

`chat_template/` - переиспользуемый минимальный шаблон веб-приложения для задач с AI-чатом. Используй его как стартовую точку, когда отдельной задаче нужен небольшой браузерный чат.

Текущий стек шаблона:

- Vite
- TypeScript
- без React
- без backend-сервера
- прямые запросы из браузера в API AI-провайдера

Держи шаблон маленьким и без лишних зависимостей. Не добавляй фреймворки, UI kits, state managers, routers или новые build tools, если задача явно этого не требует.

## AI-провайдер

Текущий провайдер - DeepSeek, используется OpenAI-compatible chat completions API.

Код, специфичный для провайдера, находится в `chat_template/src/deepseek.ts`. Если позже появятся другие провайдеры, изолируй их в небольших provider-модулях, а не размазывай API-вызовы по UI-коду.

Значения env по умолчанию для DeepSeek описаны в `chat_template/.env.example`.

## Секреты

Никогда не коммить реальные API keys, tokens, credentials или локальную конфигурацию провайдеров.

Для секретов используй локальные env-файлы:

- `.env.local`
- `.env`
- `.env.*`

В репозитории должны храниться только example-файлы:

- `.env.example`
- `.env.*.example`

Перед завершением изменений, которые затрагивают конфигурацию провайдера, при необходимости проверь, что секреты игнорируются через `git check-ignore`.

## Ожидания по `.gitignore`

Корневой `.gitignore` должен покрывать и Kotlin/Gradle, и web app артефакты.

Ожидаемые игнорируемые web-артефакты:

- `node_modules/`
- `dist/`
- `.vite/`
- `coverage/`
- `playwright-report/`
- `test-results/`
- npm/yarn/pnpm debug logs
- `*.tsbuildinfo`

Не коммить сгенерированный build output из `chat_template/dist`.

## Работа с задачами

Когда создаешь новую задачу на основе chat template:

1. Скопируй `chat_template/` в новую папку задачи, если задаче нужно отдельное независимое приложение.
2. Держи task-specific изменения внутри этой папки задачи.
3. Сохраняй `chat_template/` как чистую переиспользуемую базу, если пользователь явно не просит улучшить общий шаблон.
4. Task-specific prompts, UI experiments и provider settings клади в папку задачи, а не в общий шаблон.

## Проверка

Для изменений в `chat_template/` запускай:

```bash
npm run build
```

из папки `chat_template/`.

Когда меняется UI-поведение, запускай Vite dev server:

```bash
npm run dev
```

Затем проверь минимум:

- desktop layout
- mobile layout
- отсутствие horizontal overflow
- empty chat state
- no-key warning path
- message send path, если доступен реальный ключ

Для базовой проверки шаблона реальный API key не обязателен.

## Стиль

Всю документацию веди на русском языке. Исключения: названия технологий, пути, команды, функции, имена файлов, API-термины и другой технический текст, который лучше оставить в оригинальном виде.

UI держи минимальным, читаемым и сфокусированным на задаче. Не добавляй landing pages, marketing sections, декоративный UI, большие design systems или несвязанные product features.

Предпочитай понятные TypeScript-типы и маленькие модули. Держи `src/main.ts` сфокусированным на UI state и DOM events. Детали API-запросов держи в provider-модулях.
